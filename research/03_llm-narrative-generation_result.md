# 大規模言語モデルを用いたリアルタイム・インタラクティブ・ナラティブとマーダーミステリー・シナリオ自動生成の技術的体系

大規模言語モデル（LLM）の急速な進化は、従来の静的な脚本に基づくゲームデザインを、動的で創発的な物語体験へと変貌させている。特に、プレイヤーが自身のキャラクターや設定を定義し、それに基づいてAIが即興的にシナリオを構築、進行させる「マーダーミステリー（マダミス）」の自動生成は、計算ナラティブ学における最前線の課題である。4〜7人のマルチプレイヤー環境において、リアルタイム通信（WebSocket）を介し、一貫性を保ちながら「結末の決まっていない」物語を紡ぎ出すためには、高度なマルチエージェント・オーケストレーション、ワールドステート管理、および遅延最適化技術の統合が不可欠となる。本報告書では、これらの要求事項を満たすための技術的アプローチ、アルゴリズム設計、および推奨アーキテクチャについて包括的に論じる。

## 1. インタラクティブナラティブの技術的アプローチ

インタラクティブナラティブとは、プレイヤーの介入によって物語の展開が動的に変化する形式を指す。LLMをこの領域に適用する際、単一のプロンプトによる生成では長期間の一貫性や論理的整合性を維持することが困難であるため、複数の構造的パターンが提唱されている。

### 物語生成の主要なアーキテクチャパターン

物語生成のアーキテクチャは、自由度と制御のトレードオフに基づいて分類される。初期のシステムでは単純なプロンプトチェーンが用いられていたが、複雑なマルチプレイヤー環境では、役割を分担させたエージェント方式や、状態遷移を明示的に管理する方式が主流となっている。

| アーキテクチャパターン | 特徴とメカニズム | メリット | デメリット |
| --- | --- | --- | --- |
| プロンプトチェーン方式 | 前のLLMの出力を次の入力に順次供給する 。 | 実装が単純で、逐次的な処理に適している。 | 記憶の忘却や文脈のドリフトが発生しやすい。 |
| ステートマシン + LLM | ゲームのフェーズ（導入、調査、推理、結末）を決定論的に管理し、内容をLLMが生成する 。 | 進行の確実性が高く、ゲームバランスを制御しやすい。 | 自由度が制限され、想定外の展開に対応しにくい。 |
| マルチエージェント方式 | ディレクター、キャラクター、ナレーターなどの役割を持つ複数のエージェントが自律的に相互作用する 。 | 複雑な社会関係や、各キャラクターの秘匿情報の管理に優れる。 | 通信コストと遅延が増大し、オーケストレーションが複雑。 |
| RAG（検索拡張生成）併用方式 | 膨大な設定資料や過去の対話履歴をベクトルデータベースに保存し、必要に応じて抽出する 。 | 膨大な文脈を扱いつつ、幻覚（Hallucination）を抑制できる。 | 検索の精度に依存し、物語の「流れ」を壊す可能性がある。 |

現代的なアプローチでは、これらを統合した「ディレクター・アクター・パラダイム」が有効視されている。ディレクターエージェントが全体的な物語の緊張感やペースを管理し、アクターエージェントが個別のキャラクターの心理状態や知識に基づいて行動を決定する構成である 。この階層構造により、物語の全体的な目標（殺人の解決）と個々のプレイヤーの自由なロールプレイを両立させることが可能になる。  

### 物語の一貫性（コヒーレンス）を保つための技術

マーダーミステリーにおいて、論理的一貫性の欠如は致命的な欠陥となる。LLMが生成するテキストにおける矛盾を検出し、修正するためには、情報の構造化と検証プロセスが不可欠である。

一貫性管理の中心となるのは「ワールドステート（世界状態）」の管理である。これは物理的な位置関係、アイテムの所有権、キャラクター間の好感度や秘密の共有状況を保持するデータベースとして機能する 。特に「mistery\_murder\_ia」プロジェクトでは、知識グラフを用いて物語内の全ての変数を追跡し、ReAct（Reasoning and Acting）エージェント技術を導入して矛盾が検出された場合に該当箇所を再生成する仕組みを採用している 。  

キャラクターの記憶管理においては、短期的な「ワーキングメモリ」と長期的な「エピソディックバッファ」、そして要約された「物語アーカイブ」を使い分ける階層的メモリシステムが提案されている 。これにより、LLMのコンテキストウィンドウの制限を克服し、数時間にわたるゲームプレイ中も過去の発言との矛盾を避けることができる。また、伏線の管理については、生成パイプラインの初期段階で「ゴールドファクト（確実な事実）」を定義し、それを物語の各チャンクに埋め込むことで、論理的に解決可能なミステリーを構築する手法が有効である 。  

### マルチプレイヤー環境での物語分岐と情報管理

4〜7人のプレイヤーが参加する環境では、情報の非対称性がゲームの核心となる。各プレイヤーは自身のキャラクターのみが知る「秘匿情報」を持っており、LLMはこれを選択的に開示・隠蔽する必要がある。

| 情報管理の課題 | 技術的解決策 | 実装のポイント |
| --- | --- | --- |
| 個別情報の提供 | プレイヤーごとにフィルタリングされたコンテキストを構築する 。 | LLMへの入力に、そのプレイヤーの「視点」と「所持知識」のみを含める。 |
| 秘密情報の保護 | エージェントのプロンプト内に「開示不可」のメタデータを付与する 。 | RAG等で情報を取得する際、権限レベルに基づいたアクセス制御を行う。 |
| 同時性の処理 | WebSocketによるメッセージのキューイングとシリアライズを行う 。 | 複数のプレイヤーが同時に行動した場合、サーバー側で順序を確定させる。 |

行動の順序性については、サーバー・オーソリテーティブ（サーバー権威）モデルが推奨される。プレイヤーからの入力はWebSocket経由で一旦サーバーに集約され、物語の整合性をチェックした上で、LLMが「誰が何をしたか、その結果周囲に何が見えたか」を決定し、全クライアントへ同期配信する仕組みである 。  

## 2. シナリオ自動生成の具体的手法

マーダーミステリーの面白さは、綿密に組まれた人間関係とトリック、そして意外性のある動機に依存する。これらをLLMによってゼロから自動生成するためのパイプライン設計が重要となる。

### シナリオ構築パイプラインの設計

高品質なシナリオ生成は、単一の出力ではなく、段階的な洗練プロセスを通じて達成される 。一般的な生成パイプラインは以下のステップを踏む。  

1.  **概要と前提の生成:** プレイヤーが入力した設定（例：19世紀の英国、サイバーパンクな日本）を元に、殺人事件の基本コンセプトを決定する 。  

2.  **役割とMMOの割り当て:** 各キャラクターに対し、「Motive（動機）」、「Means（手段）」、「Opportunity（機会）」を定義する 。この際、一人が真犯人となり、他のプレイヤーには疑わしい過去や隠し事（レッド・ハリング）を付与する。  

3.  **関係性とタイムラインの構築:** キャラクター間の愛憎、借金、血縁などの関係性をマトリックス化し、事件発生前後の物理的な行動表を時間軸に沿って作成する 。  

4.  **トリックと証拠の生成:** 物理的なトリック（例：密室、時刻表の穴）を生成し、それを証明または反証するための証拠品を配置する 。  

5.  **個別台本の出力:** 各プレイヤーに対し、自分の行動指針、アリバイ、そして「他の誰の、どの秘密を知っているか」を記載したシートを生成する 。  

このプロセスにおいて、特に「論理的整合性ツリー」の構築が推奨される 。これは、証拠Aと証拠Bがあれば犯人Cを特定できるという論理構造をグラフとして保持し、物語がこのツリーに沿って展開されるように制限をかける手法である。  

### テンプレートベース vs 完全自由生成の役割分担

シナリオのクオリティを安定させるために、約50パターン程度のシナリオ構造テンプレート（例：クローズド・サークル、倒叙ミステリー、交換殺人など）を用意することが現実的な解となる 。  

テンプレートは「物語の骨組み（幹）」を定義し、LLMはその「詳細（花や葉）」を生成する。この「Orchid」モデルでは、ノードを「蕾（Bud）」として定義し、プレイヤーとの対話によってその内容を「開花（Blossom）」させる比喩が用いられる 。  

テンプレートとLLMの境界を「移行条件（Transition Conditions）」として定義することで、プレイヤーの自由な発言を許容しつつ、物語が作者（またはシステム）が意図した方向へ進むように誘導することが可能になる 。  

### 品質保証と論理的整合性チェック

生成されたシナリオが「解ける」ものであることを保証するためには、生成後に別のLLMインスタンス（探偵エージェント）に解かせてみる「セルフ・デバッグ」が有効である 。  

論理的整合性チェックの数式的な表現として、物語の整合性（Coherence）C と驚き（Surprise）S のトレードオフがある 。  

C∝∑logical conflicts1​,S∝−log(P(outcome∣clues))

高品質なミステリーは、C を最大化しつつ、最終的な解決の確率 P が低くなるように設計されなければならない。これを実現するために、MuSR（Multi-step Reasoning）のようなデータセット作成手法に基づき、事実関係の推論ツリーが正しく機能しているかを生成時に検証する 。  

## 3. リアルタイムゲーム進行でのLLM活用

ゲームが開始された後は、LLMは「ゲームマスター（GM）」として、リアルタイムに変化する状況に対応し、物語をゴールへと導く役割を担う。

### ゲームマスター（GM）としての設計パターン

GMとしてのLLMは、受動的な応答ではなく、能動的な物語の進行管理を行う必要がある 。  

-   **ナレーションとNPCの統合:** プレイヤーの行動を受け取り、それを魅力的な情景描写へと変換する。同時に、プレイヤーが話しかけていないNPCの自律的な行動を生成し、世界に「生きている感覚」を与える 。  

-   **イベントトリガーと介入:** 議論が停滞した際や、重要な証拠が見落とされている際、LLMは新しいイベント（例：第二の殺人、停電、NPCの乱入）を発生させて場を活性化させる 。  

-   **段階的ヒントの提供:** プレイヤーの推理レベルを監視し、MECE（漏れなくダブりなく）の原則に基づいた質問を投げかけることで、プレイヤーを正解へと導く 。  

### レスポンスタイムとパフォーマンスの最適化

リアルタイム性を重視する WebSocket 通信において、LLMの生成待機時間はユーザー体験（UX）を著しく損なう。これを最小化するためには、以下の技術スタックの活用が求められる 。  

| 最適化手法 | メカニズム | 効果  |
| --- | --- | --- |
| ストリーミング配信 | 全文生成を待たずに、生成されたトークンから順次クライアントへ送信する。 | TTFT（Time to First Token）を劇的に短縮し、ユーザーの待ち時間を減らす。 |
| プロンプトキャッシュ | システムプロンプトや不変の世界設定のKVキャッシュを再利用する 。 | 推論コストを最大90%削減し、処理速度を向上させる。 |
| 継続的バッチ処理 | 異なるプレイヤーからの複数のリクエストを効率的にGPUへ詰め込む 。 | スループットを最大23倍向上させ、サーバーコストを抑える。 |
| 投機的実行 / プリフェッチ | プレイヤーが次に取るであろう行動（例：隣の部屋へ移動）を予測して事前に生成しておく 。 | 実際にプレイヤーが行動した際の応答時間をゼロに近づける。 |

特に、WebSocket環境下では、SSE（Server-Sent Events）とは異なり双方向の割り込みが可能であるため、プレイヤーが「入力を開始した」瞬間に生成を中断したり、先行してコンテキストを準備したりする戦略が可能となる 。  

### マルチプレイヤー同期とコンフリクト解決

複数のプレイヤーが同時に発言や行動を行った場合、物語の分岐が複雑化する。これには「PlaySocketJS」で採用されているような、CRDT（Conflict-free Replicated Data Types）にインスパイアされた状態管理手法が有効である 。  

各プレイヤーの入力は、タイムスタンプと共にサーバーへ送信され、サーバー側の「オーケストレーター」がそれらを一つの物語の流れに統合する。例えば、二人のプレイヤーが同時に同じナイフを拾おうとした場合、サーバーは先着順（または敏捷性パラメータなどのゲーム内論理）で成功者を決定し、その結果をLLMに入力して「一人がナイフを手に取り、もう一人がそれに驚く」といった描写を生成させる 。  

## 4. 「結末未確定」のゲーム設計とLLM

本プロジェクトの最大の特徴である「プレイ中の行動によって結末が動的に決定される」設計は、従来の分岐型シナリオとは本質的に異なる。

### 動的エンディング決定アルゴリズム

結末を事前に確定させないためには、物語を一種の「確率的な収束プロセス」として捉える必要がある。プレイヤーの行動履歴から、犯人が逃げ切る確率 Pescape​ と、捕まる確率 Pcapture​ をリアルタイムにスコアリングする 。  

P(outcome)\=σ(ω1​⋅EvidenceCount+ω2​⋅SuspicionLevel+ω3​⋅DebateSkill)

ここで、σ はシグモイド関数、ω は各要素の重みである。LLMはこれらの数値を直接扱うのではなく、「現在の状況に基づき、最も劇的で、かつ論理的に許容される結末を生成せよ」という高レベルの指示を受ける 。DiriGentフレームワークのように、エージェントの「理想の世界（犯人が逃げたい）」と「現実の世界（証拠が見つかった）」の間の緊張感（Tension）を分析し、その緊張が解消される方向へと物語を誘導する手法が推奨される 。  

### ゲームバランスの動的調整（DGB）

参加プレイヤーの熟練度や参加度に応じて、難易度を自動調整する機能が必要である。

-   **沈黙プレイヤーへの対応:** 特定のプレイヤーが発言していない場合、LLMが操作するNPCがそのプレイヤーに直接質問を振る、あるいはそのプレイヤーにしか見えない重要な証拠を提示する 。  

-   **難易度のスケーリング:** 推理が早すぎる場合は、新たな謎や嘘の証拠（Red Herring）を投入して混乱させ、逆に遅すぎる場合は、段階的に直接的なヒントを提供する 。

## 5. 先行事例と論文

LLMを活用したインタラクティブ・ナラティブの分野には、多くの先行事例と学術的知見が存在する。

### 業界の主要な事例

-   **AI Dungeon / NovelAI:** 初期のLLMゲーム。自由度は高いが、長期間の整合性管理に課題があった 。  

-   **Character.AI:** キャラクターの個性（Persona）の再現に特化したプラットフォーム。マダミスにおけるNPC制御の参考になる。

-   **Inworld AI / Convai:** ゲームエンジン（Unity/Unreal）と統合可能なNPC制御AI。WebSocketベースの低遅延通信を実現している 。  

-   **ウズ (UZU) / これからミステリー:** 日本国内の主要なマダミスアプリ。アプリ側で進行を管理し、一部でAI生成やリアルタイム監修を取り入れている 。  

### 学術研究とオープンソース

-   **Jubensha研究:** 中国のミステリーゲーム「劇本殺（Jubensha）」をターゲットにしたマルチエージェント・インタラクション・フレームワークの研究が進んでいる 。  

-   **MuSR (Multi-step Reasoning):** LLMが複数ステップの論理推論（特にミステリー）をどのように解くか、または生成するかを評価するためのデータセットと手法 。  

-   **Drama Llama / Orchid:** インタラクティブ・ストーリーテリングにおける作者の意図とAIの自律性のバランスを制御するためのフレームワーク 。  

## 6. 推奨アーキテクチャ

以上の調査を踏まえ、4〜7人参加型のリアルタイム・マダミスアプリに最適なアーキテクチャを提案する。

### 提案するアーキテクチャ構成

システムは、ステートフルな通信を維持しつつ、重いLLM処理を並列化・高速化するための階層構造を持つべきである。

1.  **通信レイヤー (WebSocket Server):**

    -   Node.js または Python (FastAPI) を使用し、AWS EC2やRenderなどの「サーバーフル」な環境でホストする 。  

    -   各ルームの状態（参加者、進行フェーズ）を管理する。

2.  **オーケストレーション・レイヤー (LangGraph / CrewAI):**

    -   メインのゲームループをグラフ形式で管理し、各エージェントの行動順序を制御する 。  

    -   ディレクターエージェントが「次の展開」を決定し、アクターエージェントが「台詞」を生成する。

3.  **データ・永続化レイヤー (Redis & Vector DB):**

    -   Redisを用いてリアルタイムの世界状態（World State）とセッション文脈を保持する 。  

    -   Vector DB (Pinecone, Weaviate等) を用いて、プレイヤーごとの秘密設定や過去の事実をRAG形式で提供する 。  

4.  **AI推論エンジン (LLM Selection):**

    -   **シナリオ生成時:** 論理的思考に優れた GPT-4o または Claude 3.5 Sonnet を使用し、一貫性のある設定を構築する 。  

    -   **リアルタイム進行時:** 低遅延・低コストの Gemini 2.0 Flash または GPT-4o-mini を使用し、ストリーミングレスポンスを行う 。

### 結論と展望

大規模言語モデルを用いた次世代のマーダーミステリーアプリ開発において、鍵となるのは「論理的な堅牢性」と「劇的な柔軟性」の高度なバランスである。MuSRのような多段階推論フレームワークを活用してシナリオの整合性を担保しつつ、WebSocketによるリアルタイム通信とプロンプトキャッシュによる遅延最適化を実装することで、従来のパッケージ型ゲームでは不可能だった「プレイヤーと共に作り上げる、二度と繰り返せない物語体験」を提供することが可能になる。今後は、プレイヤーの感情分析（感情を汲み取った表情や音色の生成）や、マルチモーダルな証拠提示（AIによる画像・音声証拠のリアルタイム生成）との統合が、さらに没入感を高めるための重要な技術領域となるだろう。
